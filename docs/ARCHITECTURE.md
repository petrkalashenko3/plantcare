# 🏗️ Архитектура проекта «Plantata»

Документ для защиты: что за проект, на чём сделан, как устроен внутри, какой
файл за что отвечает и что с чем связано.

---

## 1. Что это

**Plantata** — веб-приложение-помощник по уходу за комнатными растениями:

- **Справочник** растений с карточками ухода (полив, свет, пересадка, ядовитость, особенности);
- **Личная коллекция** «Мои растения» со своими заметками, датами и параметрами полива;
- **Избранное**;
- **Напоминания** о поливе (в интерфейсе + браузерные уведомления).

Соответствует базовой версии кейса «Помощник по уходу за растениями».

---

## 2. Технологический стек и почему он такой

| Слой            | Технология                        | Почему так                                                        |
|-----------------|-----------------------------------|-------------------------------------------------------------------|
| Разметка        | HTML5                             | —                                                                 |
| Стили           | CSS3 (+ CSS-переменные)           | Цвета/отступы в одном месте (`:root`), легко менять тему          |
| Логика          | Чистый JavaScript (ваниль)        | Без фреймворков и сборщиков — код читается построчно, легко защищать |
| Хранение данных | LocalStorage (браузер)            | По ТЗ данные хранятся локально, без аккаунта и сервера            |
| Справочник      | JSON-файл (`data/plants.json`)    | Статические данные «внутри приложения»; готовый формат под будущую БД |

**Никакого бэкенда и базы данных в базовой версии нет** — это требование кейса
(«данные можно хранить локально в браузере или внутри приложения»). Серверная
часть — это усложнение, к которому текущая структура готова (см. раздел 12).

---

## 3. Как запустить

`fetch` (загрузка `data/plants.json`) работает только по HTTP, поэтому нужен
любой локальный статический веб-сервер. Самый короткий способ:

```bash
python -m http.server 8000
```

Затем открыть **http://localhost:8000**. Терминал держать открытым — пока он
работает, работает и сервер.

> ⚠️ При изменении кода браузер кэширует старые `.js`/`.css`. После правок жать
> **Ctrl+Shift+R** (жёсткая перезагрузка), либо держать открытым DevTools (F12) →
> Network → галочка **Disable cache**.

## 5. Модель данных (три сущности)

**1. Растение справочника** (`data/plants.json`) — только чтение:

```json
{
  "id": "monstera",         // уникальный код, связывает справочник с коллекцией
  "name": "Монстера",
  "emoji": "🌿",
  "light": "…",             // освещение
  "watering": "…",          // рекомендации по поливу (текст)
  "repotting": "…",         // пересадка
  "toxicity": "…",          // ядовитость (текст для показа)
  "toxic": true,            // ядовитость (флаг для логики/фильтров/значка)
  "features": "…"           // особенности
}
```

**2. Избранное** (LocalStorage, ключ `plantcare:favorites`) — просто массив id:

```json
["monstera", "aloe-vera"]
```

**3. Растение коллекции** (LocalStorage, ключ `plantcare:my-plants`):

```json
{
  "id": "mp_xxx",              // уникальный id записи
  "plantId": "monstera",       // ССЫЛКА на карточку справочника (по ТЗ)
  "customName": "Монстера на кухне",
  "addedDate": "2026-07-04",   // дата добавления
  "lastWateredDate": "2026-07-04",
  "wateringIntervalDays": 5,   // параметр напоминания
  "notes": "Южное окно"
}
```

> **Про ключи `plantcare:`** — это внутренний идентификатор хранилища. При
> ребрендинге на «Plantata» мы намеренно его не меняли, чтобы не потерять уже
> сохранённые данные пользователя. Это нормальная практика.

---

## 6. Хранение данных

- **Справочник** грузится один раз через `fetch('data/plants.json')` и кэшируется
  в памяти (переменная `plantsCache` в `catalog.js`).
- **Пользовательские данные** (избранное, коллекция) лежат в **LocalStorage** —
  это встроенное хранилище браузера «ключ→значение» на стороне пользователя.
  Данные переживают перезагрузку и не зависят от сервера. Так как LocalStorage
  хранит только строки, объекты/массивы сериализуем через `JSON.stringify` и
  читаем через `JSON.parse` (это делает `storage.js`).

## 7. Разбор файлов (что внутри и с чем связано)

### `index.html`
Разметка приложения:
- шапка и `nav` с тремя кнопками-вкладками (`data-tab`);
- три `<section>`-вкладки: `#tab-catalog`, `#tab-my-plants`, `#tab-favorites`;
- **модалка деталей** `#plant-modal` (заполняется из `plant-details.js`);
- **модалка-форма** `#myplant-modal` с `<form id="myplant-form">` (добавление/редактирование);
- подключение всех скриптов в конце `<body>` (порядок важен — см. раздел 8).

### `css/style.css`
Все стили. Вверху `:root` с CSS-переменными (цвета, радиусы, тени). Дальше блоки
по компонентам: шапка, навигация, карточки, модалки, форма, статусы напоминаний.

### `js/utils.js` — утилиты
Не зависит ни от чего. Даёт всем модулям:
- `todayISO()` — сегодняшняя дата `YYYY-MM-DD` по **локальному** времени;
- `formatDate(iso)` — в вид `ДД.ММ.ГГГГ`;
- `addDaysISO(iso, days)` и `daysBetweenISO(a, b)` — арифметика дат в **UTC**
  (чтобы не «съезжало» из-за часового пояса);
- `generateId()` — уникальный id для записи коллекции.

### `js/storage.js` — хранилище
Единственный файл, который напрямую трогает LocalStorage. Даёт:
- `readJson/writeJson` — общие помощники;
- избранное: `getFavorites`, `isFavorite`, `toggleFavorite`;
- коллекция: `getMyPlants`, `getMyPlant`, `addMyPlant`, `updateMyPlant`, `removeMyPlant`.

Зависит от: `utils.generateId` (для новых записей).

### `js/catalog.js` — справочник
- `loadPlants()` — `fetch` + кэш `plantsCache`;
- `getPlantById(id)` — поиск растения (используется везде, где нужно раскрыть id в данные);
- `createPlantCard(plant)` — DOM-карточка (переиспользуется в «Избранном»);
- `renderCatalog(plants)` — сетка карточек справочника.

### `js/plant-details.js` — детальная карточка
- `renderPlantDetails(plant)` — HTML всех рекомендаций + кнопки ⭐ и «в мои растения»;
- `openPlantDetails(id)` / `closePlantDetails()` — показать/скрыть модалку;
- `toggleCurrentFavorite()` — переключить избранное для открытого растения.

Зависит от: `catalog.getPlantById`, `storage.isFavorite/toggleFavorite`,
`favorites.renderFavorites`.

### `js/favorites.js` — избранное
- `renderFavorites()` — рисует вкладку, **переиспользуя** `catalog.createPlantCard`.

Зависит от: `storage.getFavorites`, `catalog.getPlantById/createPlantCard`.

### `js/my-plants.js` — личная коллекция
- отрисовка: `myPlantDisplayName`, `createMyPlantCard`, `renderMyPlants`;
- форма (одна на добавление и редактирование): `openAddMyPlantForm`,
  `openEditMyPlantForm`, `closeMyPlantForm`, `submitMyPlantForm`, `setFormLinkedPlant`.

Режим формы определяется по `form.dataset.editId`: есть → `updateMyPlant`, нет → `addMyPlant`.

Зависит от: `storage.*` (CRUD), `catalog.getPlantById`, `utils.todayISO/formatDate`,
`reminders.getWateringStatus/renderRemindersBanner`.

### `js/reminders.js` — напоминания
- `getWateringStatus(item)` — статус полива (просрочен/сегодня/через N дней);
- `getDuePlants()` — кому пора полив;
- `renderRemindersBanner()` — сводка сверху вкладки «Мои растения»;
- `waterPlant(id)` — кнопка «Полил» (ставит `lastWateredDate = сегодня`);
- уведомления: `notificationsSupported`, `notifyDuePlants`, `enableNotifications`.

Зависит от: `utils.addDaysISO/daysBetweenISO/todayISO`, `storage.getMyPlants/updateMyPlant`,
`my-plants.renderMyPlants/myPlantDisplayName`.

### `js/app.js` — точка входа
Связывает всё вместе:
- `switchTab/initNavigation` — переключение вкладок;
- `initPlantDetails` — клики по карточкам (делегирование) и кнопки в модалке;
- `initMyPlants` — форма, кнопки «Полил»/«редактировать»/«удалить», кнопка уведомлений;
- `initCatalog` — загрузка справочника;
- на `DOMContentLoaded` запускает инициализацию в правильном порядке.

## 8. Порядок загрузки скриптов (и почему)

```html
utils → storage → catalog → plant-details → favorites → my-plants → reminders → app
```

Скрипты подключены обычными `<script>` без модулей, функции лежат в глобальной
области. Порядок идёт «от простого к сложному»: сначала утилиты и хранилище, в
конце — `app.js`, который всё оркеструет. Вся реальная работа запускается на
`DOMContentLoaded`, когда DOM и все скрипты уже готовы.

Важный нюанс: справочник грузится асинхронно (`fetch`), поэтому в `app.js`
разделы, которым нужны данные растений, рисуются **после** `await initCatalog()`:

```js
await initCatalog();   // дождались справочника
renderFavorites();     // им нужны данные растений (эмодзи, названия)
renderMyPlants();
notifyDuePlants();
```

## 9. Ключевые сценарии (как это работает по шагам)

**Открыть карточку растения:**
`app.initPlantDetails` слушает клики на контейнере (делегирование) → по `.card`
берёт `data-plant-id` → `openPlantDetails(id)` → `getPlantById` → `renderPlantDetails`
→ модалка показывается.

**Добавить в избранное:** в модалке кнопка ⭐ → `toggleCurrentFavorite` →
`storage.toggleFavorite` → перерисовка модалки и вкладки «Избранное».

**Добавить в коллекцию:** в модалке «➕ В мои растения» → `openAddMyPlantForm(plantId)`
→ заполнение формы (дата = сегодня) → «Сохранить» → `submitMyPlantForm` →
`storage.addMyPlant` → `renderMyPlants`.

**Напоминание:** `renderMyPlants` → `createMyPlantCard` берёт `getWateringStatus`
→ на карточке статус + кнопка «Полил»; `renderRemindersBanner` показывает сводку.
Кнопка «Полил» → `waterPlant` → `updateMyPlant(lastWateredDate=сегодня)` → перерисовка.

## 10. Логика напоминаний (формула)

```
следующий полив = дата последнего полива + интервал (дней)     // addDaysISO
осталось дней    = дней между (сегодня → следующий полив)       // daysBetweenISO
```

- `< 0` → 🔴 просрочен;
- `= 0` → 💧 полить сегодня;
- `> 0` → ✅ полив через N дней.

«Пора полить» = все записи с `осталось дней ≤ 0`. Даты считаются в UTC, чтобы не
зависеть от часового пояса и перехода на летнее время.

## 11. Браузерные уведомления

Через стандартный **Web Notifications API** (чистый клиентский JS, без бэкенда):
- `Notification.requestPermission()` — запрос разрешения у пользователя;
- `new Notification(title, {body})` — показ уведомления.

Браузер сам интегрирован с центром уведомлений ОС, поэтому уведомление всплывает
как **системный тост Windows**. Срабатывает при загрузке страницы (если разрешено
и есть что полить) и по кнопке «🔔 Напоминать в браузере». Если браузер не
поддерживает API — честно сообщаем (graceful degradation).

Ограничение: пока сайт закрыт, уведомления не приходят — для этого нужен Service
Worker + серверный push (усложнение).

## 12. Приёмы, которые стоит назвать на защите

- **Разделение ответственности** — каждый модуль про одно (хранилище / справочник / детали / коллекция / напоминания).
- **Репозиторий-обёртка** над LocalStorage (`storage.js`) — вся работа с хранилищем в одном месте, легко заменить на серверный API.
- **Делегирование событий** — один обработчик на контейнер вместо обработчика на каждой карточке.
- **Переиспользование кода** — карточка растения (`createPlantCard`) одна и та же в справочнике и избранном.
- **Одна форма на добавление и редактирование** — режим по `dataset.editId`.
- **Graceful degradation** — уведомления и битые данные в хранилище не роняют приложение.
- **Стабильные ключи хранилища при ребрендинге** — не теряем данные пользователя.

## 14. Ограничения и куда расти (усложнения)

- **Сервер + БД** — вынести справочник и данные пользователя на бэкенд (Python:
  Flask/FastAPI) + БД, чтобы синхронизировать между устройствами. Текущий слой
  `storage.js` для этого и изолирован — меняется только он.
- **Фоновые/пуш-уведомления** — Service Worker + Web Push.
- **Рекомендации** — подбор растений по тегам/предпочтениям.
- **Обмен растениями + чат** — при наличии сервера.
